# Knowledge Hub 项目手册

## 设计目标

这个系统服务于长期、多学科、可迁移的学习工作流。核心目标不是堆出复杂结构，而是建立一套几年尺度上仍能运行的本地知识基础设施。

## 分层职责

- 文件系统主库：`02_library/` 保存实体资料，便于备份、迁移和跨工具访问。
- Obsidian 笔记层：`03_vault/` 管理 MOC、概念、方法、问题、项目和日常记录。
- Zotero 元数据层：管理作者、年份、出版信息、标签、批注、阅读状态和引用。
- Codex 自动化层：维护规则、模板、脚本、审计和批处理流程。

## 目录原则

- 顶层目录少而稳定。
- `02_library/<subject>/<branch>/<source_type>/` 是实体资料标准路径。
- `subject_main` 与 `subject_branch` 取自 `00_system/taxonomy/subjects.yaml`。
- `source_type` 固定为 `books`、`papers`、`courses`、`reference`。
- 项目、状态、优先级、地区、时期等变化较快的信息放到 metadata、标签或笔记正文。

## 维护节奏

### 每日

- 清理当天新增资料的最小 intake。
- 至少为一条重要资料建立或补齐 source note。
- 在 daily note 中记录输入、理解、问题和下一步。

### 每周

- 清理 `01_inbox/`。
- 补齐 `zotero_key`、作者、年份和阅读状态。
- 检查最近新增资料是否已接入 MOC。
- 跑一次 `audit_system.py`。

### 每月

- 检查目录是否失控，是否出现标签爆炸、孤儿笔记或无效链接。
- 检查模板与脚本是否仍贴合当前工作流。

## 防过时原则

这个系统的目标不是绑定某一代工具，而是让知识资产比工具寿命更长。判断一个变化是否值得引入时，优先保护“内容可迁移性”，再追求“当前使用便利性”。

### 哪些部分最不容易过时

- `02_library/` 中的实体文件主库。
- `03_vault/` 中以 Markdown 为主的笔记内容。
- `00_system/taxonomy/subjects.yaml` 中的稳定学科与分支定义。
- “路径只表达稳定维度”的组织原则。
- inbox -> library -> Zotero -> source note -> MOC 的闭环流程。

这些部分即使未来更换 Obsidian、Zotero、Codex 或其他 AI 工具，也仍然可以迁移，因为它们依赖的是本地文件、普通文本和清晰规则，而不是某个云端黑箱。

### 哪些部分最容易过时

- 与某个具体工具绑定的脚本接口。
- 只适配当前工作流的模板字段。
- 某些插件、自动化配置或第三方集成方式。
- 当前 taxonomy 尚未覆盖的新领域或新交叉方向。

这类部分要被视为“可替换层”，而不是系统本体。

### 升级时的判断顺序

遇到新技术、新工具或新工作流时，按下面顺序判断：

1. 它是在增强现有结构，还是要求推翻现有结构？
2. 它是否仍允许实体资料保留在本地文件系统？
3. 它是否仍允许笔记保留为可读的 Markdown 或其他开放格式？
4. 它是在补强检索、总结、链接能力，还是把核心资产锁进专有数据库？
5. 如果一年后不用它，退出成本是否可接受？

如果一个新工具只是提升搜索、批注、整理、总结或自动化效率，通常可以接入。

如果一个新工具要求把资料、笔记、分类规则全部迁进不可导出的封闭系统，就要谨慎。那不是升级，更像把房子的地基换成租来的。

### 推荐的演化策略

- 保持“主库在本地，增强在外围”。
- 优先增加适配层，不轻易重写内容层。
- 优先修改规则和脚本，再批量移动资料。
- 优先让新工具消费现有结构，而不是强迫现有结构迎合工具噪音。
- 定期清理已经不再使用的插件、字段和脚本入口。

换句话说：让工具围着知识资产转，不让知识资产围着工具版本号转。

### 对新技术的默认态度

- 对 AI：积极使用，但把 AI 输出视为可审阅草稿，不视为唯一真源。
- 对新笔记工具：可以试，但先从导入 `03_vault/` 的一小部分开始。
- 对新文献工具：优先检查是否支持 linked attachment 或稳定导出。
- 对新数据库式知识工具：只有在它能明确带来结构收益且退出成本低时才考虑。

### 年度回顾时要问的 5 个问题

1. 过去一年里，哪些工具变化了，但内容资产几乎没受影响？
2. 哪些脚本已经和当前工作流脱节？
3. 哪些 frontmatter 字段已经长期不用？
4. taxonomy 是否需要补充新的稳定分支？
5. 如果今天把 Obsidian 或 Zotero 换掉，最难迁移的部分是什么？

如果答案主要集中在脚本、插件和局部流程上，说明系统依然健康。

如果答案开始集中在“核心内容离不开某个专有工具”，就说明系统正在变脆，需要回到本地文件和开放格式上收权。

## 常见反模式

- 跳过 inbox，直接把文件扔进 `02_library/`。
- 把状态、项目名或优先级写进目录。
- 为一两个资料临时新增分支。
- 在 Zotero 中保存附件副本，而不是 linked attachment。
- 创建 source note 后长期不补 `zotero_key`、作者和年份。
